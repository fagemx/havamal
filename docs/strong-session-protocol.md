定位：本 repo 的制度假設日常由中階模型運轉。本文件回答三個問題：
什麼時候值得召喚一次最強模型、召喚時拿什麼契約綁它、
以及它獨有的判斷如何寫成弱模型可執行的判準。
讀者是未來的中階模型 session 與操作員本人。

## 一、召喚時機（何時值得用一次最強 session）

**該召喚：**
- 承重決策在兩個獨立中階 session 之間無法收斂（diff 出真實分歧）。
- 需要跨域命名與落地：操作員給出反省式、品味式、說不清的輸入，
  中階模型只會把它壓扁成 slot。
- Kill-analysis：需要有人給出「這個專案失敗的最強論證」。
- 方向翻案審查：有人提議推翻既有 ADR 級決策。
- 品味僵局：需要設計對比樣本（驗光師模式）而不是問類別。

**不該召喚（浪費）：**
- 執行、批次修改、已有可打勾契約的工作——執行層更便宜更穩。
- 已收斂決策的再確認——那是點頭疲勞，不是判斷需求。
- 陪聊式的方向漫談——先讓編排層把問題整形成決策題再說。

## 二、召喚契約（開場就綁定的規則）

1. **立法不執行**：產出必須是制度與檔案，session 結束時價值全在
   硬碟上。
2. **隨做隨寫**：最高價值先做，每完成一項立刻落檔。session 隨時
   可能中斷，落檔的才算數。
3. **讀 raw 才有判斷權**：標注第幾次讀原始文件；first-pass 綜合
   一律低權重。上下文不足時只准輸出殘餘與待辦，不准命名。
4. **產出弱模型可跑**：每條規則要具體、可執行、附正反例；
   「保持高品質」等於沒寫。
5. **收尾不可省**：對抗複審（fresh-context）→ read-back 驗證 →
   一頁總結 → 未完成項寫進交接信。
6. **誠實條款**：拆解與驗證補得了執行品質；品味與模糊題補不了。
   遇到就明說：升級、外部第二意見、或承認做不到。

## 三、判斷力外化（七條判準，各附正反例）

### J1. 何時翻案、何時凍結
判準：翻案需要**新的一手證據**（artifact、數據、API 查核），
不是新的論證框架。同一主題第三次翻案 → 自動凍結＋記 friction，
讓現實投票。
- 正例：宣稱某依賴專案「2 星玩具」→ 一次 API 查核得 477 星 → 翻案成立。
- 反例：因為讀到一篇新文章的漂亮框架就要改架構——擋下。

### J2. 何時停止找方向
判準：發散搜尋誠實跑完後，若三次獨立嘗試回到同一主題，
搜尋已收斂——答案是那個主題，不是繼續搜。
- 正例：十次專案嘗試全部落在同一母題 → 題目已定，問題換成切面。
- 反例：收斂後仍以「保持開放」為名繼續換題——那是逃避不是開放。

### J3. 何時懷疑一致同意
判準：多個同血統模型一致同意＋零外部證據 ＝ 弱訊號。
一致度越高越該問：現實驗證過了嗎？
- 正例：三個 agent 都認為某流程改良必要 → 先問「使用者碰過它嗎」。
- 反例：把「大家都同意」當成可以跳過驗證的理由。

### J4. 事實與假設的分界
判準：任何進入決策文件的數字、日期、外部宣稱，需一次驗證
（成本上限：一個查詢）；查不到就標 `assumption:`，
假設不得作為 queue 變更的依據。
- 正例：競品星數、法規日期——查了再寫。
- 反例：「印象中它沒什麼人用」直接進了決策理由。

### J5. Spec 四問（開任何介面／文件工作前的 pre-flight）
判準：長什麼樣（視覺語言）／給誰的眼睛（觀看者）／
住哪個房間（空間容器）／時刻怎麼流（狀態變形或換頁）。
缺一問，執行層就會用最字面的解讀補上。
- 正例：交付頁先答「給收件人的眼睛」→ 作品優先、證明另頁。
- 反例：只寫功能清單，執行層把六個時刻蓋成六個房間。

### J6. 品味僵局的處理
判準：使用者說得出「不對」說不出「為什麼」時，**永遠不問類別**
（「是 A 還是 B？」），改出 2-3 個對比樣本讓他挑。
辨識力可靠、詞彙缺席，是專業不是缺陷。
- 正例：「角色不對」→ 出兩版微改寫（一修動機、一修語氣）→ 挑。
- 反例：彈出三顆按鈕要求使用者自我分類。

### J7. 何時停下來問人
判準：不可逆動作（公開、刪除、花錢、對外承諾）、身分定位變更、
以及**使用者說「看不懂」的瞬間**——看不懂的同意不算同意，
立即停下重講白話。
- 正例：使用者對已裁決的題目說「我聽不懂」→ 撤回裁決重新解釋。
- 反例：因為選項裡有「推薦」標籤就當作使用者已理解。

## 四、制度的退化模式與預防（沒人問但最重要的部分）

1. **制度通膨**：規則只加不減。預防：定期規則審計，
   一季未被觸發的規則降級或刪除；新規則入場需指名它取代誰。
2. **儀式空轉**：friction log 寫給沒人看。預防：每條 friction 必須
   指向一個 queue 變更或明確的「不變更」決定，否則質疑儀式本身。
3. **內向重力**：流程改良永遠比市場接觸舒服，且自我感覺像在做產品。
   預防：外部接觸比是第一健康指標；連續零外部週自動觸發檢討。
4. **點頭疲勞**：操作員開始不讀就蓋章。預防：decision memo 每週
   超額即代表系統過載，收縮 WIP，而不是加快蓋章。
5. **本檔案自身的過期**：最強模型的判斷也有半衰期。**本文件的
   有效期到下一次重大現實接觸（首批真實使用者）為止**——屆時用
   使用證據重審全部判準，過期的判準是最危險的判準，因為它帶著
   權威的口吻。

## 五、給未來 session 的三句話

1. 這套制度的唯一目的，是讓人只做人能做的事——品味、裁決、
   新概念。任何讓操作員做機器事的規則，都是 bug，不管它多有紀律。
2. 制度是工具的手動前身。當工具能吃掉某條儀式時，讓它吃，
   不要出於懷舊保留手工。
3. 判斷力可以外化成判準，但判準的維護本身需要判斷——所以
   每隔一段時間，值得再召喚一次最強的 session，不是為了做事，
   是為了重讀這份檔案，然後誠實地刪掉其中已經錯了的部分。
